# Dev-Kid Init-Check

Validates that the current project is correctly wired up for dev-kid orchestrate/execute. Distinct from `/devkid.init` (which scaffolds a fresh project) — this is a HEALTH check on an already-initialized project.

## What This Does

Runs 10 checks at the project root:

| # | Check | What it verifies |
|---|-------|------------------|
| 1 | dev-kid on PATH | The CLI is installed and reachable |
| 2 | dev-kid.yml | Project-level dev-kid config present |
| 3 | ralph-tiers.json | Sentinel tier configuration present |
| 4 | .env keys | At least one provider key set (Anthropic / OpenAI / Google) |
| 5 | tasks.md | Task list exists at project root (or symlink) |
| 6 | tasks.md symlink | Recommended: tasks.md is a symlink to specs/<feature>/tasks.md |
| 7 | execution_plan.json | Wave plan generated by orchestrate |
| 8 | constitution | Governance file present (.specify or memory-bank) |
| 9 | Claude Code hooks | .claude/hooks/ has expected hooks installed |
| 10 | sentinel parser | Live test: `dev-kid sentinel-run --list` actually parses tasks.md |

Reports PASS / WARN / FAIL per check.

## Usage

```bash
/devkid.init-check
```

## Exit Codes

- `0` — all PASS or WARN, project ready for `/devkid.execute`
- `1` — at least one FAIL, project not ready (output names what to fix)
- `2` — environment problem (dev-kid not installed, etc.)

## Execution

```bash
#!/bin/bash
set -e

if ! command -v dev-kid &>/dev/null; then
    echo "❌ dev-kid CLI not on PATH. Install dev-kid first."
    exit 2
fi

dev-kid init-check
```

## When to Use

- **Right after `/devkid.init`** — confirms the scaffold is complete
- **Before your first `/devkid.execute`** — don't get surprised by missing config halfway through wave 4
- **On a fresh clone of an existing project** — checks if everything carries over
- **In CI as a smoke test** — fail the build if dev-kid prerequisites aren't satisfied
- **When `/devkid.execute` does something unexpected** — narrow down the cause
