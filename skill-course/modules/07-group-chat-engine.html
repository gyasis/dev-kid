The Group Chat Engine

How those iMessage-style component conversations reveal themselves one bubble at a time

Picture a stage play where actors enter on cue

Every line of the play is written and the whole cast is already backstage, waiting in the wings. A brief hush — the rustle of curtains — tells you who's about to speak, then one actor steps into the light and says their line.

Here's the twist: the entire script is sitting in the page from the very start, completely hidden. The performance isn't about finding the words — it's about timing their reveal.

🎭
The reveal

Those chat conversations between components? Nothing is fetched, nothing is generated live. The whole exchange is pre-written and hidden in the markup — shown one bubble at a time, on cue.

Every line waits backstage — hidden and name-tagged

Each message starts with display:none, so it's fully built but invisible. And every line carries a data-sender tag — a name tag pinning it to one actor.

CODE 
<div class="chat-message" data-sender="actor-a" style="display:none">
  <div class="chat-avatar" style="background: var(--color-actor-1)">A</div>
  <div class="chat-bubble">
    <span class="chat-sender">Actor A</span>
    <p>Hey, I need the data for this item.</p>
  </div>
</div>

Notice what's already here

The avatar's color and initial, the speaker's name, the exact words — all of it is written into the line before the page even loads. The engine never invents a thing.

The engine learns the cast, then reveals on cue

Before the show, the engine walks every line and builds an actor map — a cast list pairing each name tag with that actor's initial and color. Then to reveal a line, it shows the typing hush wearing the next speaker's avatar, waits a beat, and brings them on stage.

CODE 
const actors = {};
messages.forEach(msg => {
  const sender = msg.dataset.sender;
  const avatar = $('.chat-avatar', msg);
  if (avatar && !actors[sender]) actors[sender] = { initial: avatar.textContent.trim(), style: avatar.style.background };
});
function showNext() {
  const msg = messages[index]; const sender = msg.dataset.sender;
  if (typingEl && actors[sender]) { /* show typing dots wearing sender's avatar */ typingEl.style.display = 'flex'; }
  setTimeout(() => { typingEl.style.display = 'none'; msg.style.display = 'flex'; index++; }, 800);
}

🏷️
The name tag is the only link

The engine never hard-codes "Actor A is teal." It reads data-sender off each line and looks the actor up — so adding a new speaker means adding a tagged line, nothing else.

⏱️
setTimeout is the dramatic pause

That 800-millisecond beat is the whole reason it feels human. Remove it and four bubbles slam onto the screen at once — true, but lifeless.

🎬
Reveal, never create

The lines already exist, hidden. showNext just flips one from backstage to on-stage. The "magic" is sequencing what's already written.

The life of a single bubble

Each line follows the exact same four-beat journey from the wings to the spotlight. Trace it once and you understand the whole engine.

1
Hidden in the wings

The line sits in the page with display:none — fully built, name-tagged, invisible. Waiting for its cue.

2
The typing hush appears

The engine reads the line's data-sender, dresses the typing dots in that actor's avatar and color, and shows them — "this person is about to speak."

3
The bubble steps into the light

After the 800ms pause, the hush hides and the message flips to visible — the actor delivers their line.

4
The index advances

The engine bumps its bookmark forward by one, so the next "Next Message" press reveals the following line. Repeat until the script runs out.

💡
Aha! — the data is all there; the experience is the timing

Withholding information you already have, to control pace and attention, has a name engineers use everywhere: progressive disclosure. It's the heartbeat of onboarding flows, animations, and good UI everywhere.
