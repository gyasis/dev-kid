The Flow Animation Engine

How a tiny "route" baked into one attribute drives every box-lights-up, dot-flies-between-them diagram

Think of a subway line map

The route is printed on the wall — a fixed list of stations in order. As the train arrives, each station lights up, and a little dot glides along the track to the next stop.

The map never moves. The map is just a list. All the lighting-up and gliding is done by one little engine that reads the list and walks it, station by station.

🚇
The reveal

Every animated flow diagram in this course is a subway map: a list of stops written as data, plus one engine that lights each stop and slides a dot between them.

What you just rode

That whole animation is driven by a single list of five stops, written as text inside one HTML attribute. Change the list, and the show changes — the engine never does.

The route is a list, written as text

The stops live in one attribute called data-steps. Its value is written in JSON: a list where each stop says which box to light up and what caption to show.

CODE 
<div class="flow-animation" data-steps='[
  {"highlight":"flow-actor-1","label":"User clicks the button"},
  {"highlight":"flow-actor-2","label":"Backend calls the DB","packet":true,"from":"actor-2","to":"actor-3"}
]'>
  <div class="flow-actor" id="flow-actor-1">...</div>
  <div class="flow-packet" id="flow-packet"></div>
</div>
PLAIN ENGLISH

The conductor: one step at a time

When you press Next Step, one small function called next() runs. It reads the current stop from the list and acts it out — then bumps a counter so the following press grabs the next stop.

CODE 
function next() {
  const s = stepsData[step];
  $$('.flow-actor', containerEl).forEach(a => a.classList.remove('active'));
  if (s.highlight) {
    const hEl = $('#' + s.highlight, containerEl) || $('#flow-' + s.highlight);
    if (hEl) hEl.classList.add('active');           // ← scoped to THIS flow
  }
  if (s.packet && s.from && s.to) animatePacket('flow-' + s.from, 'flow-' + s.to);
  step++;
}
PLAIN ENGLISH

Here's the conductor — run it once to advance one station.

Grab the current stop from the route list (which one we're on is tracked by step).

First, dim every box in this diagram — turn off the previous station's light.

If this stop names a box to highlight…

…find that box inside this diagram only, then fall back to a page-wide search if needed.

…and light it up. The comment says it plainly: look scoped to this flow, not the whole page.

(closes the highlight step)

If this stop also wants a dot to fly, slide it from the "from" box to the "to" box.

Bump the counter so the next press reads the next stop.

Done — that's one station, fully acted out.

📋
stepsData is the printed route

It's the list of stops the code read out of data-steps once, at the start, using JSON.parse. The engine just walks it.

🔦
"highlight" means light up a box

The stop names a box by its id. The engine adds an "active" glow to that box and removes it from the others.

🎯
$('#id', containerEl) = find it inside this diagram

This is a scoped lookup: it searches only this animation's boxes, not the whole page — so two diagrams can reuse the same box names safely.

Two silent traps every flow author hits

Because the route is just text in an attribute, two tiny mistakes can break the whole animation — with no error popup the reader ever sees. Learn to suspect the data, not the engine.

❜

The apostrophe trap

The whole route sits between single quotes: data-steps='[...]'. A stray apostrophe in a label — like the user's request — is read as the closing quote. The route ends early, JSON.parse fails, and the animation just sits there. Fix: write &apos; instead.

👯

The duplicate-id trap

Put two flows on one page that both use flow-actor-1, and a page-wide search grabs the first one. The dot — positioned with getBoundingClientRect — flies to the wrong diagram, sometimes off-screen. Fix: scope the lookup to the container.

⚠️
Why these are sneaky

Both break silently. There's no crash, no red banner on the page — the animation simply does nothing, or the dot drifts away. The first place to look is always the data inside data-steps, not the engine.

Spot the bug: which line breaks the route?

Find the line that breaks the JSON:

1 <div class="flow-animation" data-steps='[
2 {"highlight":"flow-actor-1","label":"User opens the page"},
3 {"highlight":"flow-actor-2","label":"Read the user's request"},
4 {"highlight":"flow-actor-3","label":"Send the reply"}
5 ]'>
🔎
The fix in one move

Change the user's request to the user&apos;s request. The &apos; is a safe stand-in for an apostrophe that the attribute's closing quote won't mistake for itself — exactly what the live animation at the top of this module does.

Change the route, change the show

The engine you read never knew about codebases, briefs, or agents. It only walks a list. Swap in a different list and you get a completely different animation — without touching a single line of the engine.

💡
Aha! — data-driven animation

This is the same big idea behind the quizzes: the data declares what happens, and one generic engine acts on it. Engineers call this data-driven — and when you spot it, you can tell an AI agent "keep this data-driven" to get clean, reusable code.
