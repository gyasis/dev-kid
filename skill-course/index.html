02

The Directory & build.sh

A course is a folder, not a file — and that single choice is why these courses come out consistent and cheap to generate.

Think flat-pack furniture

When you buy a flat-pack bookshelf, the hard parts arrive pre-made in the box — the panels are already cut, the screws already threaded. You don't re-mill the wood; you just bolt the panels together by following the instruction sheet.

A codebase-to-course directory works exactly the same way. The pre-made panels are the engine; the instruction sheet is one tiny shell script that assembles them.

📦
The whole idea in one line

The panels (CSS + JavaScript) never change between courses. Only the content panels — the lessons — get cut fresh. You arrange pre-made parts; you never re-make them.

What's actually in the box

Here is the course directory. Notice the three jobs each file has: pre-made (copied unchanged), customized (lightly adjusted per course), and written fresh (the actual lessons).

skill-course/ the course folder — everything lives here
styles.css 🔒 PRE-MADE · all the looks, copied unchanged every time
main.js 🔒 PRE-MADE · all the interactivity, copied unchanged every time
_base.html 🔧 CUSTOMIZED · the shell — title, accent color, top navigation
modules/ ✍️ WRITTEN FRESH · one file per lesson — this is the only part the AI authors
_footer.html 🔒 PRE-MADE · closes the page, copied unchanged
build.sh 🔒 PRE-MADE · the instruction sheet that bolts it all together
🔒
Pre-made (verbatim)

Copied byte-for-byte. The AI is told to verbatim these files — never to rewrite them. styles.css, main.js, _footer.html, build.sh.

🔧
Customized once

_base.html — only the course title and accent color get swapped per course.

✍️
Written fresh

Everything inside modules/ — the lessons. This is the only creative work the AI does.

The entire assembler — yes, really

This is the complete instruction sheet. Four lines. It doesn't build anything itself — it just glues the pre-made panels and the fresh lessons into one finished page.

CODE 
#!/bin/bash
set -e
cat _base.html modules/*.html _footer.html > index.html
echo "Built index.html — open it in your browser."
PLAIN ENGLISH

"This is a Bash script" — the line that tells the computer how to run it.

"If anything goes wrong, stop immediately" — don't push a half-built page.

Paste the shell, then every lesson, then the footer — end to end — into one finished file.

Print a friendly "done!" message so you know it worked.

The middle line is the whole magic trick. Let's unpack the three words that do all the work.

cat A built-in command that pastes files end-to-end. To concatenate means 'join into one'. (cat is short for concatenate.)
modules/*.html A glob — the * means 'every .html file in modules/'. It expands sorted, so 01-…10- land in order.
> index.html Take everything that was pasted together and write it into a new file called index.html — the page you actually open.
🔢
Why the numbers in front matter

Files are named 01-…, 02-…, 10-…. The glob sorts alphabetically, so the number prefix is what guarantees lessons assemble in the right order — no list of files to maintain by hand.

What an agent actually writes

Each lesson the AI produces is a fragment — a single <section> with no <html>, <head>, or <body> tags around it. Those tags already live in _base.html and _footer.html.

CODE 
<section class="module" id="module-2">
  ... screens, visuals, quizzes ...
</section>
PLAIN ENGLISH

Open one self-contained lesson panel and give it a unique id so navigation can jump to it.

Fill it with the actual teaching: explanations, diagrams, the quiz.

Close the panel. That's it — no page wrapper, because the page wrapper lives elsewhere.

⚠️
Why a full HTML page here would break everything

If a module included <html> tags, cat would paste ten full pages into one file — a nonsense document with ten <head>s. Fragments are required precisely because they get concatenated.

The pattern you just learned has a name

Fixed engine, swappable content. The styling and interactivity are built once and reused; only the lessons change. This is the same idea behind website themes, app plugins, and document templates.

🎨

Themes

Your blog's posts stay the same; you swap the theme to change the whole look. Fixed content, swappable engine — the same trick in reverse.

🔌

Plugins

The app's core never changes; you drop in new plugins for new abilities. Fixed engine, swappable parts.

📄

Templates

An invoice template is fixed; you only fill in this customer's numbers. Fixed shell, swappable content.

💡
Aha! — Fixed engine, swappable content

Because the AI never regenerates the engine, every course looks consistent, costs fewer tokens, and inherits fewer bugs. Recognizing this pattern helps you design AI pipelines that don't waste effort re-writing boilerplate every run.

Briefs & Parallel Agents

How ten course modules get written at the same time — by writers who never opened the codebase.

A film shoot, not a one-person crew

Ten modules of this course were written at the same time by separate AI helpers that never read the source code. The trick is a single document called a brief.

🎬
The metaphor: a film shoot with a shot list

A director scouts the location once, then hands each camera crew a single shot brief — pre-loaded with everything for that one scene. The crews shoot in parallel. Nobody re-reads the whole screenplay before every take.

🎬
The director

Reads the whole screenplay once, plans every scene — this is the main AI agent.

📋
The shot brief

One page per scene, pre-loaded with the lines, props, and angles — this is a module brief.

🎥
The camera crews

Each shoots one scene from its brief, all at once — these are the parallel writing agents.

A brief is a tidy work order. It tells the writer the teaching angle, hands over the exact code to show, and lists which reference sections to read — and nothing else. Here is the template.

The title and number of the module this writer owns.

A checklist of which interactive widgets to build — quiz, chat animation, step cards.

A note on what comes before and after, so the modules connect smoothly.

Why "pre-extracted code" is the whole game

Because the snippets are already in the brief, the writer never opens the real repo. That saves the writer's context budget for the actual writing instead of spending it reading a giant codebase it would mostly ignore.

The skill's own instructions spell out exactly what each writer is handed — and, just as importantly, what it is not. Dispatch happens in groups of up to three at a time.

Send the modules out to writers, up to three running at once. Each writer gets a small, fixed kit:

Its own brief — the work order for its one module.

The two house-style guides: how to write content, and the mistakes to avoid.

Just the slices of the design and widget manuals it actually needs — not the whole books.

The blocklist: no full codebase, no master instructions, no peeking at other writers' briefs. Less to read, less chance of drifting off-style.

Its one brief, the two style guides, and the exact reference sections it needs. A small, focused kit.

🚫

What it NEVER gets

The full codebase, the master instructions, or any other writer's brief. Anything that would bloat its memory or cause drift.

The whole pipeline, end to end

From a raw codebase to a finished course, the skill follows six steps. The first three are the director's planning; the last three are the shoot and the edit.

1
Analyze the codebase

The main agent reads the project once and figures out what it does and how the pieces fit.

2
Design the curriculum

It decides how many modules there are and what each one teaches.

3
Write the briefs

For each module it drafts a work order — pre-extracting the exact code snippets into the brief.

4
Dispatch parallel writers

It hands briefs to subagents in batches of three. They write their modules at the same time.

5
Assemble

The finished module files are stitched together into one single-page course.

6
Review

A final consistency pass checks names, accent color, and tone across every module.

When does the skill skip all this?

Briefs and parallel writers are not always worth it. For a small project, the main agent just writes the modules one after another — no work orders, no dispatching.

📝

Simple codebase

One agent writes each module in sequence. The whole thing fits in its memory, so there's nothing to pre-digest.

🎬

Complex codebase

Too big to hold at once. The agent writes briefs and dispatches parallel writers — the film-shoot pattern.

📌
The portable lesson

You can demand this pattern from any AI system: "pre-digest the inputs into self-contained work orders, then give each worker only what it needs." That is how you scale AI work without blowing the context budget.

The Navigation & Reveal Engine

As you scroll, the top bar fills, dots light up, and content fades in on cue. No magic — just one small file watching where you are on the page.

A guided gallery tour with spotlights

Picture walking through a quiet museum where the lights are off until you arrive. As you step into each room, a spotlight clicks on overhead, and a little "you-are-here" dot slides along the map by the door. Nothing is lit ahead of you — that calm, on-cue reveal is exactly what this course's page does as you scroll.

🖼️
The metaphor: the museum that lights up as you arrive

The page is the gallery. The progress bar is the path you've walked so far. The nav dots are the map by the door. And the spotlight that clicks on when you enter a room is each section fading into view — never before you get there.

📊
The path behind you — the progress bar

A thin line across the top that fills as you scroll down, showing how far through the tour you are.

🟢
The map by the door — the nav dots

One dot per module. The room you're in glows; rooms you've passed stay filled.

💡
The spotlight — reveal-on-scroll

Each section starts invisible and fades in the moment you step into view. Calm, never all-at-once.

Aha! — the page doesn't "know" what scrolling means

There is no magic sense of motion. The page just reacts to the browser reporting positions: "this element is now in view." That is the whole idea behind reactivity — watch a value, change a label, let the styling do the rest.

The watcher, in real code

This is the actual reveal engine from the course's main.js file. It sets up one watcher, points it at every section that should fade in, and tells it what to do the instant a section appears.

CODE 

const revealObserver = new IntersectionObserver(entries => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      entry.target.classList.add('visible');
      revealObserver.unobserve(entry.target);
    }
  });
}, { rootMargin: '0px 0px -8% 0px', threshold: 0.08 });

$$('.animate-in').forEach(el => revealObserver.observe(el));
PLAIN ENGLISH

Hire one watcher whose whole job is to notice when things enter the screen.

For each section the watcher reports on…

…check: is this section now actually in view?

If yes, stick a label called visible on it — that's the signal to fade in.

The one-time trick: tell the watcher to stop watching that section. It already revealed; no need to watch it again.

The settings: only count it as "in view" once it's a touch past the bottom edge, and at least 8% of it is showing.

Finally: find every section marked to animate, and put it under the watcher's eye.

Three small words carry the whole idea. An observer is the watcher itself; isIntersecting is its yes-or-no answer to "is this on screen now?"; and unobserve is how it lets go once the job is done.

Two dials on the watcher

That last line had two settings in curly braces. They decide exactly how far into the screen a section must be before it counts as "arrived." Think of them as a sensitivity dial and an edge-margin dial.

🎚️

threshold: 0.08

The threshold is the trigger point: only fire once at least 8% of the section is showing. A peek of the top edge isn't enough — wait until it's clearly arriving.

📐

rootMargin: -8% bottom

This shrinks the watched area by 8% at the bottom of the viewport, so a section reveals a beat after it pokes in — making the fade feel deliberate, not jumpy.

🔌
Where does the actual fade live?

Not in this code! The watcher only adds the visible label. The fade-in animation — going from invisible to solid — lives in the page's styling, which reacts to that label. Code changes a class; styling does the rest.

The map by the door: active vs. visited dots

Now the nav dots. A second small function runs as you scroll and decides, for each dot, whether it should glow as "you are here" or stay filled as "already visited." It works off one number: where the middle of your screen falls on the page.

CODE 

function updateNavDots() {
  const scrollMid = window.scrollY + window.innerHeight / 2;
  modules.forEach((mod, i) => {
    const dot = navDots[i];
    const top = mod.offsetTop, bottom = top + mod.offsetHeight;
    if (scrollMid >= top && scrollMid < bottom) { dot.classList.add('active'); }
    else if (window.scrollY + window.innerHeight > top) { dot.classList.add('visited'); }
  });
}
PLAIN ENGLISH

Each time we re-check the dots, do this:

Find the vertical middle of what you can see — your scroll position plus half the screen height.

Go through every module, paired with its dot…

…grab that module's dot from the map,

…and figure out where that module starts and ends on the page.

You-are-here: if the middle of your screen lands inside this module, mark its dot active — make it glow.

Already-seen: otherwise, if you've scrolled past this module's start, mark it visited — keep it filled.

📍
Aha! — "you are here" is just arithmetic

There's no fancy tracking of "which module you're reading." The page simply asks: does the middle of my screen fall between this module's top and bottom? If yes, that's the active dot. A whole feeling of "the app knows where I am" comes from one comparison of numbers.

Three jobs, one tiny file

Put it together and the engine does just three things as you scroll. None of them is complicated on its own — together they create the polished, "alive" feeling of a modern scrolling page.

1
Fill the progress bar

Measure how far down the page you are, turn it into a percentage, and stretch the top bar to match.

2
Light the right nav dot

Check where the middle of your screen falls, glow the dot for that module, and fill the dots you've passed.

3
Reveal sections on cue

A watcher fades each section in the moment it enters view — once, and only once.

There's a fourth bonus job: the arrow keys. Press the down or right arrow and the page jumps to the next module; up or left jumps back — so you can tour the gallery by keyboard, not just by scrolling.

🌐
Why you should care: "scrollytelling" is everywhere

Those news features and product pages where things slide and reveal as you scroll? Same idea — scroll position becomes label changes. Once you see it's that simple, you can ask an AI for it precisely: "reveal each section on scroll with an IntersectionObserver."

The Glossary Tooltip Engine

Museum placards on a movable arm — how a word's definition floats out to exactly where you're looking, and never gets cropped.

Hover a dashed word — feel the trick

Every dashed-underlined word in this entire course pops a definition right where you're reading. Try it now: hover DOM, getBoundingClientRect, or z-index.

Here's the deliberate part: those definitions appear cleanly even inside a dark code block — a box that's specifically built to crop anything spilling out of it. That should be impossible. This module is the magic trick explained.

🏛️
The metaphor: a museum placard on a movable arm

Picture a placard that swings out from a ceiling rail to float beside whatever exhibit you lean toward. Because it hangs from the rail — not bolted inside the glass display case — the case's walls can never hide it. Our tooltips hang from the page itself, never from the word.

All an author writes is one tag

Writing a tooltip takes zero JavaScript. You wrap a word in a span and hang the definition off it as a data-definition attribute. The engine finds every one of these automatically when the page loads.

CODE 
<span class="term"
  data-definition="A background script that runs
  even when you're not looking at the page.">
service worker</span>
PLAIN ENGLISH

Start an invisible wrapper, and tag it as a glossary "term" so the engine knows to give it a tooltip.

Stash the definition right on the word itself — a sticky note that travels with it.

The definition can be a full sentence; it just lives quietly in the markup until someone hovers.

Put the actual word here, then close the wrapper. That's the entire job — no code to wire up.

That's it. The author never touches positioning, never builds the bubble. A separate engine, running once on page load, turns every class="term" into a live, hovering placard.

The engine that places the placard

When you hover, this function decides where the bubble goes. It's the heart of the trick — three small moves that together make the placard escape any box.

CODE 
function positionTooltip(term, tip) {
  const rect = term.getBoundingClientRect();
  const tipWidth = Math.min(320, Math.max(200, window.innerWidth * 0.8));
  let left = rect.left + rect.width / 2 - tipWidth / 2;
  left = Math.max(8, Math.min(left, window.innerWidth - tipWidth - 8));
  tip.style.left = left + 'px';
  document.body.appendChild(tip);          // ← appended to BODY, not the term
  const tipHeight = tip.offsetHeight;
  if (rect.top - tipHeight - 12 < 0) {     // no room above → flip below
    tip.style.top = (rect.bottom + 8) + 'px';
    tip.classList.add('flip');
  } else {
    tip.style.top = (rect.top - tipHeight - 8) + 'px';
  }
}
PLAIN ENGLISH

When someone hovers a word, run this with the word and its empty bubble.

Ask the browser: where is this exact word on screen right now? Get back its edges in pixels.

Pick a bubble width that fits the screen — never wider than 320px, never thinner than 200px.

Aim the bubble's center at the word's center, so it sits directly above it.

Nudge it back inward if it would spill off the left or right edge of the window.

Lock in that horizontal position.

The key move: attach the bubble to the page body itself — NOT inside the word. This is what lets it escape any box the word is trapped in.

Now that it's on the page, measure how tall the bubble turned out.

Check: is there room above the word? If the bubble would poke off the top of the screen...

...then place it just below the word instead.

...and flag it "flipped" so its little arrow points up instead of down.

Otherwise, there's room — place it neatly above the word, the normal way.

Done. The placard now floats exactly where it should, mounted on the ceiling rail.

The word you just hovered to read append ran this exact code. The placard you saw was built, measured, and positioned by these eighteen lines.

Why the placard escapes the glass case

Code blocks in this course set overflow: hidden so long lines wrap neatly instead of poking out the side. That same rule is a guillotine for anything born inside the box. The fix is to never be born inside it.

❌ Bolted inside the case

If the bubble is added as a child of the word — and the word sits in a code block — the block's overflow: hidden walls slice the bubble off. You see half a definition, or nothing.

✓ Hung from the ceiling rail

Add the bubble to document.body — the outermost container — and no inner box can crop it. There are no walls between it and the edge of the screen.

But moving the bubble out of the word creates a new problem: now it doesn't know where the word was. That's the second half of the trick.

📏
position: fixed (not absolute)

Pin the bubble to the window using raw screen coordinates, so no ancestor box gets a vote in where it lands.

🎯
getBoundingClientRect() supplies the coordinates

It measures the word's live position on screen, so we can place the now-detached bubble directly over it.

🔝
z-index: 10000 keeps it on top

The bubble paints above the navigation bar, the code block, everything — so it's never hidden behind another layer.

💡
Aha! — the universal escape hatch

Two silent traps every flow author hits

Because the route is just text in an attribute, two tiny mistakes can break the whole animation — with no error popup the reader ever sees. Learn to suspect the data, not the engine.

❜

The apostrophe trap

The whole route sits between single quotes: data-steps='[...]'. A stray apostrophe in a label — like the user's request — is read as the closing quote. The route ends early, JSON.parse fails, and the animation just sits there. Fix: write &apos; instead.

👯

The duplicate-id trap

Put two flows on one page that both use flow-actor-1, and a page-wide search grabs the first one. The dot — positioned with getBoundingClientRect — flies to the wrong diagram, sometimes off-screen. Fix: scope the lookup to the container.

⚠️
Why these are sneaky

Both break silently. There's no crash, no red banner on the page — the animation simply does nothing, or the dot drifts away. The first place to look is always the data inside data-steps, not the engine.

A few real tokens from the house style

The design tokens aren’t arbitrary. Each one encodes a deliberate taste decision. Here are a handful of the real values from design-system.md, with the reasoning baked in.

--color-bg: #FAF7F2 A warm off-white — “like aged paper.” Never a cold clinical white; warmth is the whole mood.
--font-display: 'Bricolage Grotesque' A bold, geometric display face with personality — explicitly NOT Inter, Roboto, or Arial.
--color-bg-warm: #F5F0E8 A slightly warmer paper, used only on alternating modules to create visual rhythm.
--color-bg-code: #1E1E2E A deep indigo-charcoal for code blocks — even the dark surfaces are tinted, never pure black.
🌟
Why alternate the background?

Even-numbered modules sit on the cool paper; odd ones on the warm paper. The subtle alternation gives the long scroll a rhythm — your eye feels each module as a distinct room, not one endless hallway. (This module is #10 — notice it’s the cooler paper.)

The content philosophy — six rules, one per card

If the design system is how the gallery looks, the content philosophy is how every placard is written. Six rules do the work. They’re shown here the way they should be shown — as cards, not a paragraph.

👀

≥ 50% visual

Max 2–3 sentences per block; every screen at least half pictures. If you’re on a fourth sentence, draw something instead.

🎭

Metaphors first

Lead with a real-world image, then ground it in the code — and never reach for “restaurant.”

📝

Code ↔ English

Real code beside a plain-English translation always beats a paragraph about the code.

🧠

Quiz application

Ask “what would you do?” in a new situation — never “what does this word mean?”

💬

Tooltip every term

Define each technical word on first use. The vocabulary is the learning.

❓

Why before how

Answer “why should I care?” before “how does it work?” — motivation before mechanism.

THE CHECKLIST

WHY EACH ONE

Stops the page from becoming a textbook — eyes glaze on the fourth sentence.

A fresh metaphor makes a concept click; the overused one stops teaching (more on this in the quiz).

Non-technical readers learn from seeing the code and its meaning side by side, not from prose describing it.

The goal is doing something with the knowledge — not reciting it back.

A learner should never have to leave the page to Google a word. The terms are the takeaway.

Motivation comes before mechanism — nobody learns “how” if they don’t buy “why.”
