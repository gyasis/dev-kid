The Sentinel

A bouncer at the door of every checkpoint — checking IDs before any work gets past the velvet rope.

"Done" is a claim, not a fact

In the last module, the Wave Executor refused to start a new wave until every task was marked done. But there's a quieter gap: between the AI says a task is done and we save it to history, who actually checks the work?

Dev-Kid slips an inspector into that gap — one whose whole job is to try and disprove the work before it's trusted. Meet the Integration Sentinel.

🛡️
Hold this picture: a bouncer at the door

Before anyone gets past the velvet rope, the bouncer checks IDs — no fakes, no troublemakers, no one over the room's limit. The Sentinel is that bouncer, and the velvet rope is the git commit that locks the wave into history.

Where the bouncer stands

The Sentinel isn't a separate stage you run by hand — it's wedged into the pipeline you already know. The moment a task is finished, the Sentinel inspects it. Only if it approves does the wave move toward its checkpoint.

If the bouncer turns the work away, the whole wave is halted — nothing gets committed until a human looks. That's the difference between "the AI claims it's done" and "this is proven done."

What the bouncer actually checks

Every time a task finishes, the Sentinel runs the same fixed routine — seven steps, always in this order. Each step catches a specific flavor of "looks done but isn't."

🔍

1 · Placeholder scan

Hunts for fake code — leftover TODO, FIXME, or stubbed-out functions in real production files.

🧭

2 · Find the test command

Auto-detects how this project runs its tests (pytest, npm test, cargo test…). No tests found? It skips ahead.

🔁

3 · Fix-it test loop

Runs the tests. If they fail, a cheap local AI tries to fix it, then a smarter cloud AI if needed.

🔌

4 · Interface diff

Checks whether the task changed any interface — the public shape other code depends on.

📏

5 · Change-radius check

Did the task blow past its scope? Too many files, too many lines, a risky interface change?

🌊

6 · Cascade analysis

If a change ripples outward, it flags which other tasks might now need attention.

📝

7 · Write the manifest

Always — pass or fail — it logs exactly what happened to a manifest file.

In the code: the bouncer's checklist

Those seven steps aren't a diagram someone drew for this course — they're written, in order, right at the top of the file that runs them. This is the real comment from cli/sentinel/runner.py.

CODE 
# SentinelRunner.run() orchestrates the complete pipeline:
#   1. Placeholder scan (pre-test-loop)
#   2. Test framework detection
#   3. Tiered micro-agent test loop (Tier 1 Ollama → Tier 2 cloud)
#   4. Interface diff
#   5. Change radius evaluation
#   6. Cascade analysis
#   7. Manifest writing
PLAIN ENGLISH

One function, run(), does the entire inspection start to finish.

First, sweep for fake code before wasting time on tests.

Figure out how this particular project runs its tests.

Try the cheap local AI first, escalate to the pricey cloud AI only if needed.

Check whether any public interface quietly changed.

Measure how big the change got — files, lines, scope.

Flag any downstream tasks the change might have disturbed.

Finally, write the receipt — every single time, no exceptions.

Notice the order is deliberate: the cheapest check (scanning text for fake code) runs first, so the Sentinel never spends money on tests for work that was obviously hollow.

The bouncer gives one of three answers

After the routine, the Sentinel hands back exactly one verdict. Knowing which is which tells you instantly whether the problem is bad code or bad setup.

✅
PASS — you may enter

Every check it could run came back clean. The work is trusted, the wave can move toward its commit.

🚫
FAIL / HALT — turned away

Something real is wrong: fake code, failing tests, or a change that blew its scope. The wave stops cold until a human looks.

⏭️
SKIP — nothing to check

No test command was found, so there was nothing to run. This is a setup gap, not a sign the code is bad.

🔎
SKIP is not the same as PASS

A SKIP means the bouncer never got to check IDs — maybe there was no ID scanner at the door. If you see SKIP, the fix is usually adding a test command, not rewriting the code.

In the code: catching fake code red-handed

Here's a real gate from cli/sentinel/runner.py. The AI left placeholder code where finished code should be — and the Sentinel doesn't just note it, it pulls the emergency brake on the entire wave.

CODE 
if violations and fail_on_detect:
    print(f"      🚫 Sentinel: {len(violations)} placeholder violation(s) detected")
    result_obj.result = "FAIL"
    result_obj.should_halt_wave = True
PLAIN ENGLISH

Did we find fake code and is the project set to be strict about it?

Then announce it loudly — "🚫 found N placeholder violations."

Stamp this task's verdict as "FAIL".

should_halt_wave = True — pull the emergency brake on the whole wave, not just this one task.

That last line is the teeth. A stub in one task doesn't just fail quietly; it stops everything downstream, because shipping fake code is exactly the kind of "looks done" that the Sentinel exists to catch.

The whole point: distrust by design

💡
aha! — PASS is earned, not assumed

Most tools assume the AI did its job and move on. The Sentinel assumes the opposite: it's built to distrust the work and actively tries to disprove it. A PASS only comes after every check fails to find a problem — which is why a PASS here actually means something.

This flips a normal instinct. "No news is good news" doesn't apply — the Sentinel makes news either way, and writes it down. That's how "the AI claims done" becomes "proven done."

A curious detail in the config

Here's something odd. Dev-Kid's settings live in a config file called dev-kid.yml. And out of the box, the Sentinel — this carefully-built safety system — ships switched off.

CODE 
sentinel:
  enabled: false
  mode: auto
PLAIN ENGLISH

All the Sentinel's settings live under this heading.

enabled: false — the bouncer isn't even at the door yet. The whole inspection is turned off.

mode: auto — when it's on, it runs fully automatically without pausing for a human.

🤔
Why ship your safety system OFF?

It seems backwards to build an elaborate inspector and then hand it to people switched off. There's a real reason — a chicken-and-egg problem hiding in here. Keep this puzzle in your back pocket; we'll crack it open in Module 9.
