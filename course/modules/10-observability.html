Observability — No Black Boxes

The thesis the whole course was building toward: the goal was never zero failures. It was zero invisible failures. A crash you can see is a lesson; a crash you can't is a mystery that costs you hours.

The plane crashed. What did the black box say?

The bug in Module 9 didn't hide because it was subtle. It hid because nothing along the way said it was happening — a crash got quietly relabeled "test failed, escalate." Nobody lied; the system just never spoke up.

That's the cardinal sin of an AI system, and it's the one thing this whole course has been circling. The fix has a name, and you should learn to demand it by that name.

🛩️
Hold this picture: the flight data recorder

A plane that crashes teaches you nothing if there's no black box. But a plane wired with full telemetry turns every incident into a recording: you replay the last minutes, hear exactly which gauge went red, and you learn. The aim of safe engineering isn't a plane that never fails — it's a plane that, if it does, tells you precisely why.

That mindset has a one-word name: observability. No black boxes. Every step must be visible as what it actually is.

Fix #6: don't believe "I passed" — go check the tests yourself

This is the heart of observability in actual code, from cli/sentinel/tier_runner.py. The engine claims it passed. Instead of trusting that claim, the Sentinel runs the tests again itself and only believes a green light it produced with its own hands.

CODE 
# #6 — confirm micro-agent's success with an independent test re-run.
if final_status == "PASS":
    confirmed, confirm_tail = _run_test_command(test_cmd, Path.cwd())
    if not confirmed:
        final_status = "FAIL"
        print("      ⚠️  exited 0 but independent test re-run FAILED")
PLAIN ENGLISH

The comment says it plainly: "confirm the success with an independent re-run." Trust, but verify.

Only bother if the engine claims it passed. (If it already failed, there's nothing to double-check.)

Run the real test command again, with our own hands, in this folder. confirmed is the honest answer.

If our own re-run does not agree…

…override the engine's "PASS" with the truth: FAIL. The claim doesn't get the last word.

And — crucially — say so out loud: print the warning. A silent override would just be a different kind of black box.

🔁
Callback to Module 4: verify, don't trust

This is the exact same reflex as the checkpoint gate from Module 4 — it refused to open until it counted the [x] marks itself, instead of trusting that a wave "said" it was done. Here it's a passed test instead of a finished wave, but the principle is identical: measure the claim against reality before you act on it.

Fix #5: a $0 paid run is a red flag, not a free lunch

You met this exact trap at the end of Module 9: a paid model reporting $0. The natural instinct is to celebrate. The observable instinct is to get suspicious — and this snippet from the same file does precisely that, out loud.

CODE 
is_paid_tier = bool(artisan_model) and not artisan_model.startswith("ollama")
if is_paid_tier and tier_cost == 0.0:
    print("      ⚠️  paid model reported $0.00 cost — likely a parse miss; "
          "cumulative budget may understate spend")
PLAIN ENGLISH

First, decide: is this a tier that costs real money? (Anything that isn't the free local Ollama model.)

If it's a paid tier and yet the cost came back exactly zero — that combination is the suspicious one.

So warn, loudly: a paid run that costs nothing almost certainly means the cost number was never read correctly — a "parse miss."

And spell out the consequence: if one run's cost vanished, the running budget total is quietly too low. Better to flag it than to silently under-count spend.

💡
aha! — an anomaly is data, not noise

Observability isn't only "surface errors." It's also "be suspicious of numbers that are too good." A paid step reporting $0 is the cousin of a smoke detector reading zero in a burning room. The right response to an impossible reading is never to trust it — it's to investigate the instrument.

The Watchdog: observability for things that don't print anything

An AI agent lives in a world of text — it only knows what gets printed. But real work spawns real processes, and processes can lie in two opposite ways that a text-only agent is blind to.

👻
Orphan = the process outlives its task

The task is marked done, but the process it launched is still alive — burning CPU, holding memory, costing money. The text says "complete." Reality says "still running."

🧟
Zombie = the task outlives its process

The task still shows running, but the process behind it already died. The status board says "in progress." Reality says "nobody's home."

Both are invisible to a token-based agent — there's no text to read. The Rust Watchdog catches them because it watches real PIDs, not text — it's a flight recorder for the part of the system that never speaks.

📟
Same thesis, new sensor

Surfacing stderr makes text failures visible. Watching PIDs makes process failures visible. Different instruments, one goal: leave no corner of the system where a failure can hide unobserved.

The toolkit: five reflexes of an observable system

Everything in this module reduces to five concrete habits. These are the dials and recorders of the cockpit — the things you can demand by name from any AI tool you steer.

📣

1. Surface stderr/stdout

Don't swallow a tool's output — show the error tail. The actual crash message is the single most useful thing you can rescue from a failure.

⚠️

2. Warn on anomalies

A paid step reporting $0, a loop with 0 iterations, an impossibly fast run — treat "too good to be true" as a defect, not a win.

🔁

3. Independently verify

Never trust a single self-reported "I passed." Re-run the test yourself. The cheapest insurance against a confident lie.

📟

4. Watch real processes

See orphans and zombies the text world can't. Watch PIDs, not just printed status, so nothing runs (or dies) in silence.

🧾

5. Always write the manifest

A report on every run, even a crash — wrapped in try/finally. The flight recorder that survives the wreck.

🧾
Callback to Module 6: the manifest is always written

Card #5 is the reflex you already met. Module 6's manifest_writer.py files three documents — manifest.json, diff.patch, summary.md — on every single run, wrapped in try/finally so the report gets filed even when the run crashes. A crash that leaves a paper trail is debuggable; a crash that leaves nothing is the black box we've spent this whole course learning to refuse.

The enemy was never failure

💡
aha! — crashes and bugs are acceptable, even useful, if they're observable

The goal of all this was never zero failures — that's impossible. The goal is zero invisible failures. A crash you can see in a report is a lesson you learn in minutes. A crash that hides behind a single exit code is the Module 9 bug: it costs you months. The enemy isn't failure. It's invisibility.

🤫
Callback to Module 9: the silent escalation was the whole crime

Look back at what actually broke. The engine never even started — but a non-zero exit code got silently relabeled "tier exhausted, escalate." No fallback should ever be silent. The bug wasn't that something failed — it's that the failure was allowed to pretend to be something else.
