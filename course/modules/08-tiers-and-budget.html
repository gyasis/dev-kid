The Tier Ladder & the Money Guard

Try the free responder first. Escalate to the expensive specialist only when stuck. And keep a running tab that can slam the brakes.

Triage in an emergency room

When you walk into an ER, you don't see the brain surgeon first. You see a triage nurse — cheap, fast, free to your wallet — and only get escalated to the expensive specialist if the case is still unresolved.

The Ralph Loop's fix-it engine works exactly this way: it tries a free local responder first, climbs to a paid specialist only when stuck, and a hospital administrator watches the bill the whole time.

🏥
Cheap-first, escalate-on-need, hard cap

AI calls cost money. Dev-Kid's answer is three instincts at once: start with the cheapest responder, only call the expensive one if the case is still open, and keep a running tab that can shut the whole ward.

The ladder, written as plain config

That ladder isn't hard-coded — it lives in ralph-tiers.json. Each entry is one rung, listed cheapest first. The last rung isn't a model at all; it's a human.

CODE 

{ "tiers": [
  { "name": "all-local",  "models": { "artisan": "ollama/qwen3-coder:30b", "critic": "ollama/deepseek-r1:32b" } },
  { "name": "groq-fast-free", "models": { "artisan": "groq/llama-3.3-70b-versatile" } },
  { "name": "claude-code-handoff", "type": "handoff", "requires_marker": "allow-handoff" }
] }
PLAIN ENGLISH

A list of rungs, ordered cheapest to most expensive.

Rung 1 — all-local: two models running on your own machine. Free, so it always goes first.

Rung 2 — groq-fast-free: a fast cloud model. Cheap, used only if the local one couldn't fix it.

Rung 3 — handoff: not a model at all. The handoff tier hands the case to a human, and only if a safety marker (allow-handoff) is set.

Where this ladder gets plugged in

One running tab for the whole ward

Here's the twist that makes the money guard powerful: it doesn't track cost per task. It tracks cumulative cost across every task in a run — one shared tab for the entire shift.

⚠️
Per-task cap (the trap)

A $5 limit per task across 65 tasks = up to $325. The bill runs away while every single task looks "within budget."

✅
Cumulative cap (the guard)

One tab summed across all 65 tasks. When the total nears the limit, the brakes come on — no matter which task spends it.

💡
Why "cumulative" is the load-bearing word

The danger isn't one greedy task — it's a thousand reasonable ones. Summing the spend across all of them is the only way to catch a bill that creeps up quietly.

The money guard, in real code

This is the heart of budget_tracker.py. Two questions and one action: would this overflow?, are we already over?, and add this spend to the tab.

CODE 

def would_exceed(self, projected_cost: float) -> bool:
    """True if recording projected_cost would push cumulative past the global cap."""
    return (self._cumulative_cost + projected_cost) > self.budget_usd

def is_exhausted(self) -> bool:
    return self._cumulative_cost >= self.budget_usd

def record(self, cost_usd, duration_sec, task_id=None) -> None:
    self._cumulative_cost += float(cost_usd)
    ...
    self._persist()   # survives crashes / resumes
PLAIN ENGLISH

would_exceed — a "what if?" question, asked before spending.

Take what we've already spent, add the cost of the next rung, and check: would that go over the cap? If yes, don't take that rung.

is_exhausted — a simpler check: have we already hit or passed the cap? If so, stop entirely.

record — run after a real spend: add the cost to the cumulative tab.

Then persist it — write the tab to disk so a resumed run remembers what it already spent.

💾
Why write the tab to disk?

If the spend lived only in memory, stopping and restarting the run would reset it to $0 — a budget-reset loophole. Persisting to disk means "I already spent $8" follows the run across crashes and resumes. The cap is real, not a fresh start each time.

The early-warning siren

Before the hard stop, there's a softer signal. When spending crosses a warning threshold, the tracker speaks up — so you see the bill rising before it slams the brakes.

CODE 

return (f"⚠️  CUMULATIVE BUDGET WARNING: ${self._cumulative_cost:.2f} of "
        f"${self.budget_usd:.2f} spent across {self._task_count} sentinel(s)")
PLAIN ENGLISH

Build a warning message a human can read at a glance.

It shows how much has been spent of the total cap — e.g. "$8.40 of $10.00."

And how many sentinel(s) — task checks — that spend was spread across. The count makes "where did the money go?" answerable.

🚨
Warn early, halt late

The siren fires at a percentage of the cap (say 80%), giving you a chance to react. The hard stop only comes when the next spend would actually overflow. Two layers: a heads-up, then a wall.

Economy-first is a design value, not an afterthought

This whole module is one idea made concrete. The system would rather miss a fix than hand you a surprise bill — and that priority is wired into the code, not bolted on later.

💡
"Recall gaps are acceptable; runaway bills are not."

A free local model that occasionally can't solve a case is fine — you escalate or hand it off. But a bill that quietly climbs to $325 while every task "looks fine" is a disaster. The cheap-first ladder and the cumulative cap exist to make that trade-off the default.
