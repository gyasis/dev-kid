Meet the Cast

Dev-Kid is a small theater company — once you know each member by name, you can tell the AI exactly who to fix.

A theater company, not a black box

Before we trace anything, meet the people in the building. When something breaks, you need to know who to point the AI at.

Dev-Kid is a handful of small parts, each with one job — exactly like the crew that puts on a stage show. Naming them gives you precise vocabulary to steer the AI.

🎭
Why naming the cast matters

"The watchdog is leaking processes" lands far better than "something's stuck." When you know each part's name and job, your instructions to an AI become surgical instead of vague.

Three languages, one crew

The cast speaks three languages: a Bash front door, Python brains for the thinking parts, and a single Rust watchman. You don't need to read any of them — you just need to know who's who.

Where each member sits in the building

Every crew member maps to a real folder or file in the project. This is the actual layout from the repo — the floor plan of the theater.

dev-kid/ the whole theater
cli/dev-kid Bash CLI: routes commands (the front door)
cli/orchestrator.py turns tasks.md into waves (the playwright)
cli/wave_executor.py runs waves, enforces checkpoints (the stage manager)
cli/sentinel/ the per-task validator, 10+ modules (the critic)
rust-watchdog/ compiled process/container monitor, ~1.8MB (the night watchman)
skills/ bash workflow automations (*.sh)
memory-bank/ persistent project memory (the archive)
💡
One job each — "separation of concerns"

Notice nobody does two jobs. The playwright never reviews scenes; the critic never calls cues. Engineers call this separation of concerns — and it's why you can point the AI at one named part instead of the whole machine.

The apprentice with two faces

Here's the apprentice's name tag — the part of its setup that decides what happens when you call it. One tool, but it answers to three different names.

CODE 

{ "bin": {
  "micro-agent": "./dist/cli.mjs",       // single-file generator
  "ma":          "./dist/cli.mjs",        // alias
  "ma-loop":     "./dist/ralph-loop.mjs"  // the autonomous "Ralph Loop"
}}
PLAIN ENGLISH

This is a bin map — it tells your computer what commands this tool installs.

Type micro-agent and you get the single-file generator: it writes one file, once, and stops.

ma is just a shortcut — an alias — that points to the very same generator.

ma-loop points somewhere completely different: a file that runs an autonomous repair loop. Same tool, very different behavior.

🌱
Remember this name tag

Two faces, two file destinations. ma generates; ma-loop loops. Calling the wrong one is exactly the kind of subtle mix-up that becomes the bug we hunt down in Module 9. Tuck it away for now.

Check yourself: whose job is it?

Three things go wrong on set. For each, decide which crew member you'd point the AI at. No memorizing names — just match the symptom to the role.

Two tasks ran at the same time that should NOT have — they both edited the same file and clobbered each other. Whose grouping decision was wrong?

A task was marked complete, but its background process never got cleaned up and is still eating memory. Who's supposed to catch that?

A task claims it's done, but its code is just a placeholder stub with a 'TODO: implement this later'. Which crew member should have refused to let the show move on?
