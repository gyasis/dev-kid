Inside the Sentinel

The critic isn't one blob — it's a folder of small specialists, each running one line on a fixed inspection clipboard.

Open the folder and meet the specialists

Last module the Sentinel was the critic at the door. Open it up and you find it isn't a single inspector at all — it's a small inspection team.

Picture a building inspection with a fixed clipboard: electrical, plumbing, structural — each line is a different specialist, but they all sign the same report. The Sentinel works the same way.

Why split one critic into six files?

When a specific check misbehaves, you can point the AI at one file — "the change-radius math in cascade_analyzer.py is too strict" — instead of debugging the whole inspection at once. Small files, surgical fixes.

The cleverest check: a three-axis budget

The scope inspector is the smartest of the bunch. It doesn't ask one question — it asks three, and tripping any of them flags the task as "too big to wave through."

files Did the task touch more files than allowed? (more than 3)
lines Did it change more lines than allowed? (more than 150)
interface Did it change a public interface other code depends on? (not allowed at all)
💡
Three trip-wires, not one

A small, tidy change (1 file, 4 lines) can still trip the inspector — if those 4 lines rewrote a public interface. Size isn't the only thing that makes a change risky.

The budget check, in real code

Here's the exact snippet from cascade_analyzer.py that runs the three checks. Notice that violations is a list — you can trip more than one trip-wire at once.

CODE 

violations: list[str] = []
if files_count > self._max_files:
    violations.append("files")
if lines_total > self._max_lines:
    violations.append("lines")
if interface_changes_count > 0 and not self._allow_interface:
    violations.append("interface")
budget_exceeded = bool(violations)
PLAIN ENGLISH

Start with an empty list to collect which trip-wires got crossed.

If the task touched too many files, add "files" to the list.

If it changed too many lines, add "lines" to the list.

If it changed a public interface (and that's not allowed), add "interface" to the list.

The budget is "exceeded" if the list has anything in it — even one item.

🎯
Why a list, not a yes/no?

Because the report should say which axis tripped. "You changed 9 files and touched an interface" tells you far more than a bare "too big." The list is the diagnosis.

The allowance, written in plain config

Those limits aren't hard-coded — they live in dev-kid.yml, the project's settings file. This is the actual allowance the inspector spends against.

CODE 

change_radius:
  max_files: 3
  max_lines: 150
  allow_interface_changes: false
PLAIN ENGLISH

This is the change-radius section of the config file.

Allow at most 3 files changed before flagging.

Allow at most 150 lines changed before flagging.

Do not allow public interface changes at all — flag the task if it touches one.

🔧
You can change the dials

Working on a big, sprawling refactor on purpose? Bump max_files and max_lines, or flip allow_interface_changes to true for that branch. The inspector enforces your allowance, not a fixed one.

The report always gets filed

Here's the design choice that makes the whole Sentinel trustworthy: the clerk files the paperwork every single run — pass, fail, or even crash.

📄
manifest.json

The structured record — result, which tier ran, cost, files touched.

🩹
diff.patch

Exactly what changed on disk — a diff / patch of the before and after (empty file if nothing changed).

📝
summary.md

A human-readable note, handed to the next task so it knows what just happened.

💡
"Always write the report" — even on a crash

The manifest write is wrapped in a try/finally — programmer-speak for "do this no matter what, even if everything else blows up." So there's always a paper trail, never a silent failure. This idea — observability — is the finale of the whole course.
