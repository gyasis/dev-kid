What Dev-Kid Does

The 30,000-foot map: how a plain to-do list becomes a safe, parallel, self-checking build that your AI actually executes.

The trick you already pulled off

You wrote a plain checklist in a file called tasks.md, typed dev-kid orchestrate, then dev-kid execute.

And somehow your AI coding agent built the whole thing in coordinated batches without stepping on its own toes. This module shows you the machine behind that.

🗺️
Why start here?

Once you know the four stations your list passes through, you can tell the AI exactly where a problem lives — "the plan is wrong" vs "execution stalled" vs "a check failed." That precision is the whole point.

Think of a general contractor

You hand a contractor a messy renovation wish-list. They don't pick up a hammer themselves — they turn your list into a scheduled crew plan: who works at the same time, who has to wait on whom, and which jobs would collide if done together.

Then a building inspector signs off on each phase before the next one starts — so a hidden mistake can't quietly travel into the rest of the job.

📝
Your wish-list

"Redo the kitchen, fix the bathroom, repaint the hall" — written however you think of it.

📋
The contractor's schedule

Reshapes the wish-list into batches: plumber + painter can work together; the tiler must wait for the plumber.

👷
The foreman + inspector

Dispatches each batch, then halts until the inspector signs off — no phase starts on top of a broken one.

💡
The "aha!"

Dev-Kid doesn't write your code — it choreographs the AI that writes the code. It's the contractor and the inspector, never the hammer.

The same map, written down

The whole system is this one line. Each arrow is a hand-off; the bottom row is the plain-English name for each station.

CODE 
tasks.md  →  orchestrator.py  →  execution_plan.json  →  wave_executor.py
(your list)   (the planner)       (the schedule)          (the foreman)
PLAIN ENGLISH

tasks.md — the checklist you wrote. The only file you author by hand.

orchestrator.py — the planner. A Python module that turns your flat list into a batched schedule.

execution_plan.json — the schedule it produced, saved to disk as a structured file.

wave_executor.py — the foreman. It reads the schedule and runs the work, batch by batch.

🔌
Who pushes the buttons?

The commands you type (orchestrate, execute) hit a small CLI that simply hands your request to the right Python module. It's a switchboard, not the brains.

What one line of your list looks like

Here are two real lines from a tasks.md. Every detail on the line is a signal the system reads — let's decode it.

CODE 
- [ ] T001: Implement the auth module affecting `src/auth.py`
- [ ] T002: Add login UI affecting `src/ui/login.tsx`
PLAIN ENGLISH

- [ ] — an empty checkbox. The space means "not done yet." When the AI finishes, it changes it to [x].

T001 — the task's ID, like a job number on a work order. The planner uses it to track who depends on whom.

The plain description tells the AI what to build.

The backtick path `src/auth.py` tells the system which file the task touches — its most important clue. (Why? That's the next module.)

🎯
The file path is the magic ingredient

Because T001 touches src/auth.py and T002 touches a different file, the planner knows they can't collide — so they're safe to run at the same time. That single insight is what makes parallel building safe.

The schedule the planner hands off

After orchestrate runs, the planner saves the batched schedule to disk. You rarely read this file yourself — the foreman does — but here's its shape.

CODE 
{ "execution_plan": { "phase_id": "...", "waves": [
  { "wave_id": 1, "strategy": "PARALLEL_SWARM", "tasks": [...], "checkpoint_after": {...} }
] } }
PLAIN ENGLISH

This is one parsed schedule. Everything lives under one heading: the execution plan.

A wave is one batch of work. Wave 1 here is a "PARALLEL_SWARM" — run all its tasks at once.

tasks lists the job numbers (T001, T002…) that belong to this batch.

checkpoint_after is the inspector's sign-off — the gate that must pass before the next wave begins.

So the planner's job, in one sentence: take your flat list and group it into waves that are safe to run together — each ending in an inspection gate.

The command you type is just a switchboard

When you run a Dev-Kid command, a small Bash script does one thing: it forwards your request to the correct Python module. Here are the two hand-offs that matter for this module.

CODE 
python3 "$DEV_KID_ROOT/cli/orchestrator.py" "${py_args[@]}"
...
python3 "$DEV_KID_ROOT/cli/wave_executor.py" "${remaining_args[@]}"
PLAIN ENGLISH

When you type orchestrate, the switchboard runs the planner (orchestrator.py) and passes along your options.

The ... is just other commands in between, skipped here for clarity.

When you type execute, the switchboard runs the foreman (wave_executor.py) instead.
