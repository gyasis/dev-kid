The Integration Bug

Everything you've learned hangs on one line of wiring — and that wiring dials the wrong number. The engine has never actually started.

The engine that never turned over

You've met the Sentinel, the seven checks, the tiers, the budget guard, the Ralph Loop. All of it real, all of it built. And all of it hangs on one line of wiring — a line that calls the wrong door.

The Sentinel's engine has never actually started. Every task it ever "validated" failed the same silent way. This module is about why — and it's the single most common way real AI-coding systems break.

📇
Hold this picture: dialing the wrong number off a business card

The card lists two numbers — "Sales" and "24/7 Support." You need an after-hours fix, so you dial… Sales. It's closed. You get an automated menu that waits forever for a human who never picks up. The number was perfectly valid — it just wasn't the one that does the job you needed.

Two numbers on the card

Remember from Module 7 that micro-agent is the engine. What we glossed over: micro-agent ships two entry points. Two phone numbers, printed on the same business card.

📄
micro-agent — the "Sales" line

A single-file generator. It requires a file path. Give it none, and it assumes a human is sitting there and opens an interactive prompt.

🔁
ma-loop — the "24/7 Support" line

The actual Ralph Loop — Librarian → Artisan → Critic, with --tier-config for the model ladder. This is the one that does the after-hours fixing.

Here are both numbers, exactly as the card prints them — the bin map in micro-agent's package.json.

CODE 
{ "bin": {
  "micro-agent": "./dist/cli.mjs",       // single-file generator (what Sentinel calls)
  "ma-loop":     "./dist/ralph-loop.mjs" // the Ralph Loop with --tier-config (what it NEEDS)
}}
PLAIN ENGLISH

This little table is the business card. Each line is one command you can type.

Dial micro-agent and you reach the single-file generator. This is the number the Sentinel dials.

Dial ma-loop and you reach the real fixing loop — the one with the tier ladder. This is the number it should have dialed.

Two doors, one card. Pick the wrong one and you never reach the work.

In the code: the fork in the road

Here's the exact line where the wrong number rings through to the wrong department. This is real, from micro-agent's src/cli.ts — the very first thing it checks when you dial micro-agent.

CODE 
if (!argv._.filePath || !argv.flags.test) {
  const isValidproject = await isValidProject();
  if (!isValidproject) {
    await invalidProjectWarningMessage();
  } else {
    await interactiveMode(runOptions);   // ← asks a human; no human here
  }
  return;                                // ← runAll() (the real work) NEVER called
}
await runAll(runOptions);
PLAIN ENGLISH

!argv._.filePath means "no file path was given." The Sentinel never gives one — it speaks in flags, not file paths.

So this branch fires. It checks whether the folder looks like a real project.

If it doesn't, print a warning and stop.

If it does — drop into interactiveMode: draw a prompt and wait for a human to type. (There is no human.)

return — and here's the dagger. The function exits right here.

runAll() on the last line — the call that does the actual work — is one line too late. It is never reached.

🚪
The whole bug is that one return

A return hangs up the phone before the call can be transferred to the department that does the fixing. The line below it never runs.

Nothing in this conversation is "broken" in the way you'd expect. Both parts work. They simply don't agree on how to talk. The Sentinel speaks in flags; micro-agent was waiting to hear a file path. The whole fix loop hung up on a line that was never reached.

The evidence: three calls, three dead ends

This isn't a theory — it's reproducible. Here are three real attempts to dial micro-agent the way the Sentinel does. Watch how every single one ends in a non-zero exit code.

TERMINAL — REPRODUCTION 
# A) bare dir, no file path:
✖ TTY initialization failed: uv_tty_init returned EINVAL   → exit 1

# B) valid project, no file path:
interactiveMode() → ✖ TTY EINVAL                            → exit 1

# C) real call wrapped in `script -qfc` (fake terminal):
┌  🦺 Micro Agent  ◆ ...  (interactive prompt, waiting for a human)
                                                            → HANGS → exit 124 (timeout)
A
No file path, plain folder → instant crash

It tries to draw an interactive prompt, but there's no real keyboard attached — so it crashes on TTY setup. Exit 1.

B
Real project, still no file path → same crash

Even when the folder looks valid, it heads into interactive mode and dies the same way. Exit 1.

C
Faked a terminal to dodge the crash → it hangs instead

Wrapping it in a fake terminal stops the crash — so now it succeeds at drawing the prompt and waits… forever… for a human. Exit 124 (timeout). Worse, not better.

🔁
"It always fails the same silent way" — that's the tell

Three different setups, one identical shape of failure: never the work, always the wiring. When every attempt dies the same uninformative way, stop debugging the logic and start suspecting the contract — the unspoken agreement about how two parts talk.

And even if it did run — they don't speak the same language

Suppose we fixed the dialing problem and ma-loop actually ran. There's a second contract mismatch lying in wait: the Sentinel reads the loop's results with a regex that expects an exact wording — and micro-agent prints a different wording.

OUTPUT ↔ WHAT THE SENTINEL'S REGEX WANTS 
micro-agent prints:           Sentinel's regex wants:
  "Completed in 3 iteration(s)"  →  "Iterations: 3"   ✗ no match → 0
  "Total cost: $0.47"            →  "Cost: $0.47"      ✓ (lucky substring)
  "Duration: 2.5 minutes"        →  "Duration: 2.5s"   ✗ wrong unit → 0

Two of three numbers come back as 0 — not because the work was free or instant, but because the words didn't line up. A paid run that reports $0 isn't a free lunch; it's a parse miss. (Hold that thought — it's the heart of Module 10.)

Spot the bug: which call is wrong?

Below is how the Sentinel invokes the engine, next to how the engine is actually meant to be called. Click the line that's the mistake — the call that dials the wrong number.

Find the broken call:

1 # How micro-agent is meant to be called (single file):
2 micro-agent [flags] <file path> # needs a file path
3 # How the loop is meant to be called:
4 ma-loop run <target> --tier-config tiers.json # the Ralph Loop
5 # How the SENTINEL actually calls it:
6 micro-agent --prompt X --test Y --max-runs N # no file path, wrong binary
🔍
Two mistakes in one line

Line 6 is wrong twice over. It calls micro-agent (the single-file "Sales" line) instead of ma-loop (the "Support" line that loops). And it gives no positional argument (the file path), which is exactly what tips micro-agent into the human-waiting branch. Two parts that each work, no shared contract.

The fix: two clean numbers, dialed correctly

The repair isn't clever — it's honest about the two doors. Give each entry point one clear job, point the Sentinel at the right one, and keep the old path alive but fenced off so it can't collide with the new feature.

1
micro-agent stays the single-file line

Keep it exactly as is — for one-shot, single-file generation with a file path. Nothing wrong with the Sales desk; it just isn't who you call for the loop.

2
ma-loop is the loop line

The Ralph Loop with its tier ladder. Anything that needs "keep trying until the test passes" dials this number.

3
The Sentinel routes to ma-loop run --tier-config

Point the wiring at the right door. Now the call reaches the department that actually does after-hours fixing — engine on, finally.

4
The legacy 2-tier path is kept — but gated

The old two-tier flow still works for anyone who relies on it, but it's gated behind a switch so it can't collide with the new --tier-config feature. Nothing breaks; nothing crosses wires.

🤝
The repair is an integration contract, written down

Each side now agrees on the handshake: which binary, which arguments, which output shape. That's all an integration contract is — and getting it explicit is what turns two parts that "each work" into a system that actually works.

Debug it like an engineer

No definitions — just judgment calls. Read each situation, decide what you'd do, then check. The goal is the instinct, not the trivia.

Scenario 1

Your Sentinel reports every single task as "all tiers exhausted — escalating." Every task. The models are fine in isolation; the tests pass when you run them by hand. What's the FIRST thing you'd suspect?

Scenario 2

The bare call crashed instantly (exit 1). Someone wrapped it in a fake terminal (script -qfc) to "fix the crash." Now it doesn't crash — it hangs until it times out. Why did faking a terminal make things worse?

Scenario 3

You've found the bug: the Sentinel calls micro-agent with no file path, instead of ma-loop. What's the cleanest fix that doesn't break anyone still using the old path?

Scenario 4

Even after the dialing is fixed, the Sentinel logs Cost: $0 and Iterations: 0 for a run you know used a paid model and looped several times. The run "succeeded." Should you celebrate or investigate?

The lie hidden in a single number

💡
aha! — "exited non-zero" is NOT "the work ran and failed"

Here, a non-zero exit code meant the work never started — but the Sentinel read it as "tried and failed, escalate." Trusting one number flattened three very different stories (crash, timeout, real failure) into a single misleading verdict, and the whole bug hid behind it for months.

This is the cardinal sin of AI systems: a failure that doesn't say what it actually is. The bug wasn't subtle — it was invisible, because nothing along the way was forced to tell the truth.
