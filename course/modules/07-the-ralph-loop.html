The Ralph Loop

The engine that actually writes the fix — a persistent apprentice locksmith who keeps filing the key until the lock turns.

Step 3 had a name all along

Back in the Sentinel, step 3 quietly said "run a test-fix loop." That loop has a name — the Ralph Loop — and it is the apprentice the system hires to do the actual repair work.

Picture a persistent apprentice locksmith. You don't hand them a finished key. You hand them the lock — and you say "keep filing until it turns."

🔑
You describe success as a lock, not a key

The apprentice tries a key, tests it in the lock, learns from the way it sticks, and files again. They only call in a master locksmith if they get truly stuck. This is the single most powerful idea in the whole system.

Three hands, one workbench

The apprentice isn't one person rushing — it's three roles taking turns at the bench. Each does one job, then passes the work along.

📚
The Librarian

Gathers the context and research before any filing starts — reads the surrounding code so the fix isn't a blind guess.

🔨
The Artisan

Writes the actual code change — does the filing on the key itself.

🧐
The Critic

Reviews the change before it goes anywhere near the lock — "does this look safe to test?"

🧪
The Test

Tries the key in the lock. Red means "still stuck"; green means "it turned" — and that's the only voice that can end the loop.

🎭
Why split the work into roles?

An agent that researches, writes, AND reviews all in one breath tends to fool itself. Splitting it into an agent role for each step keeps each one honest.

The test is the prompt

Here's the "aha!": the Ralph Loop turns verification — a passing test — into the spec. You don't write the instructions for the fix. You write the test that proves the fix worked, and the loop reverse-engineers code that satisfies it.

One tool, two faces — and this is the right one

Remember the two faces planted earlier? The same package ships two different commands. The Ralph Loop is the second one — and using the wrong face is the entire bug waiting in Module 9.

📝

micro-agent

The one-shot generator. Writes a file once and stops. Not the looping engine — no apprentice, no retries.

🔁

ma-loop

This is the Ralph Loop. The persistent apprentice that runs Librarian → Artisan → Critic → Test in a cycle until the lock turns.

⚠️
Burn this in: the binary is ma-loop

The engine that writes the fix is invoked as ma-loop — not micro-agent. They live in the same package but do completely different things. Calling micro-agent when you meant ma-loop gets you a one-shot guess instead of a tested fix. Hold onto this — it's the crux of Module 9.

The apprentice's instruction card, in real code

This is the actual ma-loop run command definition from src/cli/ralph-loop.ts. Every option here is a dial on the apprentice — what to fix, how to know it's done, and when to give up.

CODE 

program
  .command('run')
  .description('Run Ralph Loop iterations for a file or objective')
  .argument('<target>', 'Target file or objective to achieve')
  .option('-o, --objective <text>', 'Explicit objective')
  .option('-t, --test <command>', 'Test command to run (e.g., "npm test")')
  .option('-i, --max-iterations <n>', 'Maximum iterations', '30')
  .option('-b, --max-budget <n>', 'Maximum cost in USD', '2.00')
  .option('--simple [n]', 'Run simple mode (Artisan+Tests) for N iters before escalating')
  .option('--full', 'Skip simple mode — Librarian→Artisan→Critic→Tests from iteration 1')
  .option('--tier-config <path>', 'JSON tier config — enables N-tier escalation mode')
PLAIN ENGLISH

This defines the run sub-command of ma-loop — the apprentice's instruction card.

A one-line description: run the loop for a file or a goal.

<target> is what to work on — a specific file or a goal written in plain words.

--objective lets you spell out the goal explicitly if the target alone isn't clear.

--test is the most important dial: the test command — how the apprentice knows the lock has turned.

--max-iterations caps how many times it can loop — default 30 rounds at the workbench.

--max-budget caps the money it can spend — default $2.00 before it must stop.

--simple starts cheap: just Artisan + Test for a few rounds, then bring in the full crew.

--full does the opposite: all four roles from round one, no warm-up.

--tier-config points at a settings file that lets it climb a ladder of stronger AIs — that's the whole next module.

Two ways the apprentice stops filing

A persistent apprentice still has to know when to put down the file. The Ralph Loop has exactly two exits — one happy, one protective.

🟢
The lock turned — success

The test command passes. The fix is real and proven. Loop exits immediately, fix in hand.

🛑
The allowance ran out — stop-loss

It hit --max-iterations (too many tries) or --max-budget (too much money). The apprentice downs tools so it can't grind forever.

🪙
Why two limits, not one?

Each loop costs both time and money. An iteration cap stops it spinning forever; a budget cap stops it quietly running up a bill. Hit either, and it stops.

Start cheap, call the master only if stuck

The apprentice doesn't phone the master locksmith for every wobbly key. --simple mode runs the cheap path first — just Artisan + Test — and only brings in the full crew (and, next module, pricier AIs) if a few rounds don't crack it.

--simple [n] Try the cheap path for N rounds first, then escalate to the full crew if still red.
--full Skip the warm-up — all four roles from round one. Use when you expect a hard fix.
--tier-config Hand it a ladder of models so it can climb to a stronger AI — the next module's whole story.
💰
Cheap-first is the thrifty default

Most failing tests are simple mistakes. Starting cheap fixes the easy ones for almost nothing, and reserves the expensive crew — and expensive models — for the rare lock that genuinely won't budge. This escalation ladder is exactly what Module 8 is about.
