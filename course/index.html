Who pushes the buttons?

The commands you type (orchestrate, execute) hit a small CLI that simply hands your request to the right Python module. It's a switchboard, not the brains.

What one line of your list looks like

Here are two real lines from a tasks.md. Every detail on the line is a signal the system reads — let's decode it.

CODE 
- [ ] T001: Implement the auth module affecting `src/auth.py`
- [ ] T002: Add login UI affecting `src/ui/login.tsx`
PLAIN ENGLISH

- [ ] — an empty checkbox. The space means "not done yet." When the AI finishes, it changes it to [x].

T001 — the task's ID, like a job number on a work order. The planner uses it to track who depends on whom.

The plain description tells the AI what to build.

The backtick path `src/auth.py` tells the system which file the task touches — its most important clue. (Why? That's the next module.)

🎯
The file path is the magic ingredient

Because T001 touches src/auth.py and T002 touches a different file, the planner knows they can't collide — so they're safe to run at the same time. That single insight is what makes parallel building safe.

The schedule the planner hands off

After orchestrate runs, the planner saves the batched schedule to disk. You rarely read this file yourself — the foreman does — but here's its shape.

CODE 
{ "execution_plan": { "phase_id": "...", "waves": [
  { "wave_id": 1, "strategy": "PARALLEL_SWARM", "tasks": [...], "checkpoint_after": {...} }
] } }
PLAIN ENGLISH

This is one parsed schedule. Everything lives under one heading: the execution plan.

A wave is one batch of work. Wave 1 here is a "PARALLEL_SWARM" — run all its tasks at once.

tasks lists the job numbers (T001, T002…) that belong to this batch.

checkpoint_after is the inspector's sign-off — the gate that must pass before the next wave begins.

So the planner's job, in one sentence: take your flat list and group it into waves that are safe to run together — each ending in an inspection gate.

03

Building Waves

How a flat checklist becomes a parallel battle plan — one real algorithm, taught gently.

A flat list, a hard question

Your tasks.md is just a stack of to-do lines. Nothing in it says which jobs can happen together and which have to wait.

So how does The Orchestrator know that T003 can run at the same moment as T001 — but T005 has to hang back? That single decision is its entire job, and you're about to watch it think.

🛫
Hold this picture: boarding a plane by zones

Everyone's headed to the same destination, but you can't all cram the aisle at once. You board in groups that don't block each other — and if two passengers want the same seat, one waits for the next group.

Zones, but for tasks

A boarding zone is a batch of passengers who can move without elbowing each other. Dev-Kid calls each batch a wave — and everything in one wave runs in parallel.

🛫
One zone = one wave

A group of jobs that don't block each other, all dispatched at once.

💺
Same seat = same file

Two tasks that touch the same file can't share a zone — one must wait.

⏭️
Bumped to the next zone

A blocked task slides into the earliest later wave where nothing conflicts.

The Orchestrator's whole skill is reading your list and deciding who gets to board together. To do that, it first builds a map of who-waits-on-whom.

Three signals build the map

Before it can group anything, the Orchestrator builds a dependency graph — a who-waits-on-whom map. It gathers three kinds of clues:

🔒

File locks

Two tasks naming the same `file.py` can't ride the same wave. This is the one you control — just by how you write the task.

🔗

Explicit deps

You literally wrote "after T001" or "depends on T456" in the task. The planner takes you at your word.

🚧

## Wave N headers

A heading in the file draws a hard line. Everything below it waits for everything above it. A clean way to force order.

💡
aha! — file-lock detection is just string matching

There's no magic. The planner literally scans your task text for paths wrapped in backticks and compares the strings. That's why the `backtick` convention matters — it's the signal the machine is actually looking for. Forget the backticks and the file becomes invisible to it.

"Earliest seat that fits" — the greedy rule

How does the planner decide which wave each task lands in? It uses a greedy algorithm: for every task, drop it into the earliest wave where nothing it depends on is still pending and no file it touches is taken.

1
Take the next task in line

Go through tasks in order — no clever reshuffling.

2
Ask: what does it wait on?

Check the dependency graph and the files it touches.

3
Seat it in the earliest safe wave

The first zone where nothing conflicts. Done — move to the next task.

"Greedy" sounds reckless, but here it means predictable: same list in, same waves out, every time. No surprises is exactly what you want from a build plan.

In the code: a header draws a hard line

This is the real snippet from cli/orchestrator.py that handles a ## Wave N heading. When the planner hits one of these headers, it closes off the current group and opens a brand-new one.

CODE 
# Wave-section phase header — `## Wave N`, `### Phase 2`, etc.
if self.WAVE_HEADER_RE.match(line):
    if current_task_lines:
        self._process_task(current_task_lines, task_id, current_wave_idx)
        task_id += 1
        current_task_lines = []
    self._wave_phases.append([])
    current_wave_idx = len(self._wave_phases) - 1
    continue
PLAIN ENGLISH

Reminder of which lines this catches — a heading like "## Wave 2".

Did we hit a wave heading? (regex checks if the line matches the header shape.)

...finish that task and file it into the current wave.

So a single line of plain text in your task file becomes a hard phase boundary. That's the cheapest way to say "do this part dead last."

In the code: wiring the wait-on arrows

Once the waves exist, the planner connects them. Every task in Wave N gets an edge — a "waits-on" arrow — pointing back to every task in the earlier waves. That's what guarantees the boarding zones stay in order.

CODE 
for wave_idx, task_ids in enumerate(self._wave_phases):
    if wave_idx == 0:
        continue  # Wave 1 has no upstream phase
    for tid in task_ids:
        for prior_wave in self._wave_phases[:wave_idx]:
            for prior_tid in prior_wave:
                if prior_tid != tid:
                    graph[tid].add(prior_tid)
PLAIN ENGLISH

Go through the waves one at a time, numbered 0, 1, 2...

Is this the very first wave?

Then skip it — Wave 1 has nothing before it to wait on.

For each task sitting in this wave...

...look back at every earlier wave...

...and at every task that lived in those earlier waves.

As long as it isn't the task itself...

...record an arrow: "this task waits on that earlier one." (That's adding an edge to the graph.)

🧠
Why every task gets an arrow to the whole prior wave

The planner doesn't try to be clever about which earlier task matters. It says: "if you're in Wave 2, you wait for all of Wave 1." That blunt rule is what makes a ## Wave header a guaranteed, no-exceptions barrier.

Why this hands you the steering wheel

Here's the payoff: the plan is built from how you phrase tasks. You don't have to touch the code to change the outcome — you change the words.

✅
Wrap file paths in backticks

Write affecting `src/auth.py` and the planner can't miss the file lock.

🚧
Drop a ## Wave header to force "go last"

Anything below it waits for everything above. Cheapest way to sequence work.

🔎
When parallel tasks collide, blame the plan

If two jobs clobbered the same file, the phrasing was wrong — not the engine.

⚠️
The silent failure to watch for

Forget the backticks and the planner treats two same-file tasks as unrelated. They board the same wave, run at once, and overwrite each other's work — with no error. The fix isn't in the code; it's three backtick characters in your task line.

04

The Checkpoint Contract

The one rule that stops an over-eager AI from claiming "done" before it's actually done.

A wave finished... or did it?

A wave of tasks just ran. The AI announces it's all wrapped up and asks to start the next batch. But Dev-Kid does something unusual here — it refuses to take the AI's word for it.

Instead it stops, re-reads the task file, and demands proof. That single refusal is the safety mechanism that keeps an excitable assistant honest.

🚪
Hold this picture: a lock on a canal

A boat can't drift into the next chamber until the gate confirms the current one is filled to the right water level. No "close enough" — the level gets measured, then the gate opens. Anything less and the boat runs aground.

The gate measures before it opens

In a real canal lock, the gate doesn't trust the boat captain's say-so. It checks the water itself. Dev-Kid's Wave Executor plays exactly that role between every batch of work.

📋
The water level = tasks.md

The one source of truth the gate actually measures, line by line.

📏
The measurement = verification

Is every task in this wave really marked done? Yes or no — no maybes.

🔒
The gate opening = a git checkpoint

Only once the level checks out does the work get sealed and the next chamber unlock.

The Executor is what engineers call verification-gated: progress is only ever claimed when it's proven on disk.

In the code: the safety reflex

When the measurement comes back wrong, the Executor doesn't soldier on hopefully. It prints a loud, unmistakable line and stops — this is the entire safety reflex in one statement.

CODE 
print(f"      ❌ Sentinel HALT: {msg}")
PLAIN ENGLISH

Shout it where a human will see it: "HALT" plus the exact reason something didn't check out. No silent shrug — the gate freezes loudly so you know to look.

That ❌ Sentinel HALT line is the cheapest moment in the whole system. The AI claimed victory; the file disagreed; you caught the gap before it cost you anything.

The contract, end to end

Every wave runs the same four-step deal. Either all four happen, or the gate slams shut at step one.

1
Verify

Re-read tasks.md. Is every task in this wave marked [x]? Pass or HALT.

2
Update progress

If verified, record what just finished in progress.md — the running log.

3
Commit a checkpoint

Seal the wave's work as a git commit — a permanent, rewindable point.

4
Open the next chamber

Only now does the next wave begin. The gate moved exactly one step.

⚠️
No silent failures

The Executor would rather stop loudly than continue on a guess. A halt isn't a bug — it's the system telling you the truth out loud the moment it diverges from the AI's claim. (Keep this idea in your pocket; it's the whole spirit of the last module on watching the machine work.)

Why seal after every wave, not just at the end?

It would seem tidier to commit once when everything's finished. But a per-wave checkpoint is what makes a mistake cheap to undo.

⏮️

Rewind precisely

If Wave 3 goes sideways, you reload from the end of Wave 2 — not from scratch. Every gate is a save slot.

🧱

Proven, brick by brick

Each commit is a wave that provably passed. The history reads as a chain of verified facts, not hopeful claims.

🛟

Survives a crash

If the session dies mid-build, the last sealed wave is already on disk. You lose at most one wave, never the whole run.

The Sentinel

A bouncer at the door of every checkpoint — checking IDs before any work gets past the velvet rope.

"Done" is a claim, not a fact

In the last module, the Wave Executor refused to start a new wave until every task was marked done. But there's a quieter gap: between the AI says a task is done and we save it to history, who actually checks the work?

Dev-Kid slips an inspector into that gap — one whose whole job is to try and disprove the work before it's trusted. Meet the Integration Sentinel.

🛡️
Hold this picture: a bouncer at the door

Before anyone gets past the velvet rope, the bouncer checks IDs — no fakes, no troublemakers, no one over the room's limit. The Sentinel is that bouncer, and the velvet rope is the git commit that locks the wave into history.

Where the bouncer stands

The Sentinel isn't a separate stage you run by hand — it's wedged into the pipeline you already know. The moment a task is finished, the Sentinel inspects it. Only if it approves does the wave move toward its checkpoint.

What the bouncer actually checks

Every time a task finishes, the Sentinel runs the same fixed routine — seven steps, always in this order. Each step catches a specific flavor of "looks done but isn't."

🔍

1 · Placeholder scan

Hunts for fake code — leftover TODO, FIXME, or stubbed-out functions in real production files.

🧭

2 · Find the test command

Auto-detects how this project runs its tests (pytest, npm test, cargo test…). No tests found? It skips ahead.

🔁

3 · Fix-it test loop

Runs the tests. If they fail, a cheap local AI tries to fix it, then a smarter cloud AI if needed.

🔌

4 · Interface diff

Checks whether the task changed any interface — the public shape other code depends on.

📏

5 · Change-radius check

Did the task blow past its scope? Too many files, too many lines, a risky interface change?

🌊

6 · Cascade analysis

If a change ripples outward, it flags which other tasks might now need attention.

📝

7 · Write the manifest

Always — pass or fail — it logs exactly what happened to a manifest file.

Here's the twist that makes the money guard powerful: it doesn't track cost per task. It tracks cumulative cost across every task in a run — one shared tab for the entire shift.

Per-task cap (the trap)

Why "cumulative" is the load-bearing word

The danger isn't one greedy task — it's a thousand reasonable ones. Summing the spend across all of them is the only way to catch a bill that creeps up quietly.

The money guard, in real code

This is the heart of budget_tracker.py. Two questions and one action: would this overflow?, are we already over?, and add this spend to the tab.

CODE 

def would_exceed(self, projected_cost: float) -> bool:
    """True if recording projected_cost would push cumulative past the global cap."""
    return (self._cumulative_cost + projected_cost) > self.budget_usd

def is_exhausted(self) -> bool:
    return self._cumulative_cost >= self.budget_usd

def record(self, cost_usd, duration_sec, task_id=None) -> None:
    self._cumulative_cost += float(cost_usd)
    ...
    self._persist()   # survives crashes / resumes
PLAIN ENGLISH

would_exceed — a "what if?" question, asked before spending.

Take what we've already spent, add the cost of the next rung, and check: would that go over the cap? If yes, don't take that rung.

is_exhausted — a simpler check: have we already hit or passed the cap? If so, stop entirely.

record — run after a real spend: add the cost to the cumulative tab.

Then persist it — write the tab to disk so a resumed run remembers what it already spent.

💾
Why write the tab to disk?

If the spend lived only in memory, stopping and restarting the run would reset it to $0 — a budget-reset loophole. Persisting to disk means "I already spent $8" follows the run across crashes and resumes. The cap is real, not a fresh start each time.

In the code: the fork in the road

Here's the exact line where the wrong number rings through to the wrong department. This is real, from micro-agent's src/cli.ts — the very first thing it checks when you dial micro-agent.

CODE 
if (!argv._.filePath || !argv.flags.test) {

PLAIN ENGLISH

!argv._.filePath means "no file path was given." The Sentinel never gives one — it speaks in flags, not file paths.

So this branch fires. It checks whether the folder looks like a real project.

If it doesn't, print a warning and stop.

If it does — drop into interactiveMode: draw a prompt and wait for a human to type. (There is no human.)

return — and here's the dagger. The function exits right here.

runAll() on the last line — the call that does the actual work — is one line too late. It is never reached.

The fix: two clean numbers, dialed correctly

The repair isn't clever — it's honest about the two doors. Give each entry point one clear job, point the Sentinel at the right one, and keep the old path alive but fenced off so it can't collide with the new feature.

1
micro-agent stays the single-file line

Keep it exactly as is — for one-shot, single-file generation with a file path. Nothing wrong with the Sales desk; it just isn't who you call for the loop.

2
ma-loop is the loop line

The Ralph Loop with its tier ladder. Anything that needs "keep trying until the test passes" dials this number.

3
The Sentinel routes to ma-loop run --tier-config

Point the wiring at the right door. Now the call reaches the department that actually does after-hours fixing — engine on, finally.

4
The legacy 2-tier path is kept — but gated

The old two-tier flow still works for anyone who relies on it, but it's gated behind a switch so it can't collide with the new --tier-config feature. Nothing breaks; nothing crosses wires.

The repair is an integration contract, written down

Each side now agrees on the handshake: which binary, which arguments, which output shape. That's all an integration contract is — and getting it explicit is what turns two parts that "each work" into a system that actually works.

Fix #6: don't believe "I passed" — go check the tests yourself

This is the heart of observability in actual code, from cli/sentinel/tier_runner.py. The engine claims it passed. Instead of trusting that claim, the Sentinel runs the tests again itself and only believes a green light it produced with its own hands.

CODE 
# #6 — confirm micro-agent's success with an independent test re-run.
if final_status == "PASS":
    confirmed, confirm_tail = _run_test_command(test_cmd, Path.cwd())
    if not confirmed:
        final_status = "FAIL"
        print("      ⚠️  exited 0 but independent test re-run FAILED")
PLAIN ENGLISH

The comment says it plainly: "confirm the success with an independent re-run." Trust, but verify.

Only bother if the engine claims it passed. (If it already failed, there's nothing to double-check.)

Run the real test command again, with our own hands, in this folder. confirmed is the honest answer.

If our own re-run does not agree…

…override the engine's "PASS" with the truth: FAIL. The claim doesn't get the last word.

And — crucially — say so out loud: print the warning. A silent override would just be a different kind of black box.

The Watchdog: observability for things that don't print anything

An AI agent lives in a world of text — it only knows what gets printed. But real work spawns real processes, and processes can lie in two opposite ways that a text-only agent is blind to.

👻
Orphan = the process outlives its task

The task is marked done, but the process it launched is still alive — burning CPU, holding memory, costing money. The text says "complete." Reality says "still running."

🧟
Zombie = the task outlives its process

The task still shows running, but the process behind it already died. The status board says "in progress." Reality says "nobody's home."

Both are invisible to a token-based agent — there's no text to read. The Rust Watchdog catches them because it watches real PIDs, not text — it's a flight recorder for the part of the system that never speaks.

📟
Same thesis, new sensor

Surfacing stderr makes text failures visible. Watching PIDs makes process failures visible. Different instruments, one goal: leave no corner of the system where a failure can hide unobserved.

The toolkit: five reflexes of an observable system

Everything in this module reduces to five concrete habits. These are the dials and recorders of the cockpit — the things you can demand by name from any AI tool you steer.

📣

1. Surface stderr/stdout

Don't swallow a tool's output — show the error tail. The actual crash message is the single most useful thing you can rescue from a failure.

⚠️

2. Warn on anomalies

A paid step reporting $0, a loop with 0 iterations, an impossibly fast run — treat "too good to be true" as a defect, not a win.

🔁

3. Independently verify

Never trust a single self-reported "I passed." Re-run the test yourself. The cheapest insurance against a confident lie.

📟

4. Watch real processes

See orphans and zombies the text world can't. Watch PIDs, not just printed status, so nothing runs (or dies) in silence.

🧾

5. Always write the manifest

A report on every run, even a crash — wrapped in try/finally. The flight recorder that survives the wreck.
